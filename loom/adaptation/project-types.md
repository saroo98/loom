# Project-shape adaptation

Loom adapts planning artifacts to the work instead of treating every request as a web app.

| Shape | Emphasize | Usually skip unless triggered |
|---|---|---|
| CLI/library | command/API contracts, exit/stream behavior, packaging, compatibility | visual UI plan |
| Mobile | lifecycle, permissions, offline state, accessibility, device/store verification | SEO |
| Data/ETL/ML | lineage, schemas, idempotency, leakage, drift, reproducibility | responsive layout |
| Accounting | double entry, precision, audit trail, reconciliation, dated jurisdiction rules | generic growth metrics |
| Real-time 3D | units/coordinates, asset pipeline, spatial UX, frame/GPU budgets, real-device profiles | document breakpoints |
| Firmware/hardware | board revision, timing, power, memory, update/recovery, bench evidence | browser assumptions |
| Research/non-software | question, sources, method, uncertainty, reproducibility, deliverable format | implementation work orders that have no consumer |

Project shape modifies emphasis; it never weakens the epistemic or safety floor. Unknown domains
must produce a bounded domain-discovery artifact naming invariants, failure modes, authoritative
sources, and a real verification medium before G1. Loom must say coverage is unknown rather than
pretend a generic software checklist is domain expertise.

Only load adapters selected for the current request/project evidence. Do not load dormant or
wrong-domain rules "just in case".
