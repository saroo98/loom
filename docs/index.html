01 / WHAT ACTUALLY HAPPENS

One request in.
Six decisions behind it.

The owner does not operate this pipeline. Loom does. It asks only when a consequential unknown cannot be resolved safely.

request world domain plan proof

The output is not a longer prompt. It is a content-bound plan that can become stale, fail review, or be refused.

Read the compact kernel
  1. 01

    RESOLVE

    Find the real project.

    Identity, installation, repository root, lifecycle, and authority are established before planning begins.

  2. 02

    SURVEY

    Bind to the current world.

    Committed, staged, unstaged, untracked, runtime, and time-drifted state become one planning baseline.

  3. 03

    ROUTE

    Match rigor to consequence.

    A tiny edit stays compact. A risky migration receives the artifacts, gates, and dependency-aware work orders it needs.

  4. 04

    DISCOVER

    Use the domain's rules.

    If Loom lacks coverage, it says so, finds the affected invariants and current facts, and blocks authorization until the evidence applies.

  5. 05

    SEAL

    Make planning precede building.

    The artifact set, G1 review, work-order topology, touched paths, and acceptance evidence become tamper-evident.

  6. 06

    VERIFY

    Close the loop in the real medium.

    Target-device, browser, ledger, pipeline, hardware, or other real evidence replaces a host-authored “passed” claim.

02 / THE USEFUL PART

Loom knows when
not to continue.

Planning is trustworthy only when uncertainty changes behavior. These are stops, not warnings hidden in a report.

IFLOOM DOESWHY
The repository changed during planningBLOCKS G1

The sealed plan no longer describes the same world.

The domain or governing rule is unknownDISCOVERS, THEN RE-GATES

Generic software habits cannot stand in for domain authority.

The work-order plan changed after approvalREFUSES EXECUTION

Approval belongs to exact content, not a filename.

Nothing changed or real evidence is missingREFUSES COMPLETION

Pre-existing work and host-authored status cannot receive plan credit.

04 / LEARNING WITH AN EXIT

Useful knowledge stays.
Dead weight leaves.

Loom does not keep one growing prompt. It separates transferable judgment, domain knowledge, project facts, and installation state. Each partition has its own loading and retirement rules.

  • GENERAL

    Calibration and decision habits can transfer when evidence says they do.

  • DOMAIN

    3D, accounting, firmware, and other rules load only for matching work.

  • PROJECT

    Repo facts remain with the exact project lineage.

  • FORGOTTEN

    Content is erased; a bounded tombstone prevents resurrection.
